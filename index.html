Bem-vindo ao Curso de Controle de Versão!

Olá! Se você colabora com projetos de software, seja na criação de requisitos, na validação de funcionalidades ou na leitura de documentação, este curso é para você. Vamos desmistificar o "mundo do código" e mostrar como o controle de versão organiza o trabalho de todo o time, incluindo o seu.

Usaremos analogias do universo da mineração para tornar tudo mais claro e prático. Ao final, você se sentirá muito mais à vontade para interagir com as equipes de desenvolvimento usando ferramentas como o GitHub.

Módulo 1: O Problema - Por que precisamos organizar o trabalho?

Aula 1: O caos dos "arquivos finais"

Imagine a seguinte situação: a equipe de geologia precisa atualizar um mapa de sondagem crucial para uma nova fase de exploração. Surgem arquivos como mapa_final.docx, mapa_final_agora_vai.docx, mapa_versao_Joao.docx. Qual é o correto? Esse é o problema da falta de um "ponto central da verdade". Trabalhar com cópias de arquivos gera confusão, retrabalho e um alto risco de usar informações desatualizadas.

Aula 2: O que é Controle de Versão?

Para resolver esse caos, usamos um Sistema de Controle de Versão (VCS). Pense nele como um "livro de registros geológico". Cada alteração é registrada, com data, autor e descrição. Nada é perdido, apenas novas camadas são adicionadas. O sistema mais usado no mundo para isso se chama Git.

Módulo 2: A Solução - Conhecendo as Ferramentas

Aula 3: O "Armazém Central" e as "Cópias Locais" (Repositórios)

O Repositório Remoto (no GitHub) é o "arquivo técnico central". Cada colaborador pega uma cópia para trabalhar em sua mesa, o Repositório Local, sem alterar o original diretamente.

Aula 4: Apresentando o Git e o GitHub

Git é a ferramenta que você instala em seu computador para controlar as versões. GitHub é a plataforma online, a "sala de projetos", onde hospedamos os repositórios para colaborar em equipe.

Módulo 3: Na Prática - Colaborando em um Projeto

Aula 5: O Mapa do Projeto - Entendendo o GitFlow

O GitFlow é o "mapa de tráfego" do projeto. Ele define branches com propósitos claros: main para produção, develop para desenvolvimento, e feature para novas funcionalidades. Seu trabalho sempre começará em uma branch feature que nasce da develop.

Aula 6: Fazendo sua cópia e sugerindo mudanças (Fork e Branch)

O processo é seguro: primeiro, você faz um Fork (sua cópia do projeto no GitHub). Depois, dentro do seu projeto, você cria uma Branch (uma ramificação da develop) para trabalhar isoladamente na sua mudança.

Aula 7: Salvando e Descrevendo seu Trabalho (Commit e Conventional Commits)

Cada "ponto de salvamento" é um commit. Para padronizar, usamos o Conventional Commits, com uma mensagem clara como feat(requisitos): adiciona critério de aceitação. Isso cria um histórico limpo e compreensível.

Aula 8: Enviando suas sugestões para o time (Push e Pull Request)

Após fazer seus commits, você faz um Push para enviar sua branch ao GitHub. Em seguida, abre um Pull Request (PR), que é o pedido formal para que sua contribuição seja revisada e integrada à branch develop.

Aula 9: Entendendo as Versões do Software (Versionamento Semântico - SemVer)

O Versionamento Semântico (SemVer) usa o formato MAJOR.MINOR.PATCH (ex: 1.4.2) para comunicar o tipo de mudança em uma nova versão. MAJOR para quebras de compatibilidade, MINOR para novas funcionalidades, e PATCH para correções.
