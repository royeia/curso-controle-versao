Bem-vindo ao Curso de Controle de Versão!

Olá! Se você colabora com projetos de software, seja na criação de requisitos, na validação de funcionalidades ou na leitura de documentação, este curso é para você. Vamos desmistificar o "mundo do código" e mostrar como o controle de versão organiza o trabalho de todo o time, incluindo o seu.

Usaremos analogias do universo da mineração para tornar tudo mais claro e prático. Ao final, você se sentirá muito mais à vontade para interagir com as equipes de desenvolvimento usando ferramentas como o GitHub.

Módulo 1: O Problema - Por que precisamos organizar o trabalho?

Aula 1: O caos dos "arquivos finais"

Imagine a seguinte situação: a equipe de geologia precisa atualizar um mapa de sondagem crucial para uma nova fase de exploração.

  • Ana, a geóloga líder, cria o arquivo mapa_sondagem_v1.docx
  • João, especialista em geofísica, revisa e salva como mapa_sondagem_revisado_Joao.docx
  • Carla, da equipe de planejamento, adiciona notas sobre a operação e salva como mapa_sondagem_FINAL.docx
  • No dia seguinte, Ana faz uma última correção e envia por e-mail o arquivo mapa_sondagem_FINAL_agora_vai.docx

Qual é a versão correta? A de Carla ou a última de Ana? O que João alterou foi incluído? Esse é o problema da falta de um "ponto central da verdade". Trabalhar com cópias de arquivos em pastas e e-mails gera confusão, retrabalho e um alto risco de usar informações desatualizadas, o que em um projeto de mineração pode ter consequências sérias.

⚠️ Problemas comuns sem controle de versão:
  • Perda de trabalho quando alguém sobrescreve o arquivo errado
  • Impossibilidade de saber quem fez qual alteração
  • Dificuldade em reverter mudanças problemáticas
  • Conflitos quando duas pessoas editam o mesmo arquivo simultaneamente
  • Falta de rastreabilidade e auditoria

Aula 2: O que é Controle de Versão?

Para resolver esse caos, usamos um Sistema de Controle de Versão (VCS - Version Control System). Pense nele como um "livro de registros geológico" ou um diário de bordo de uma perfuração.

Analogia: Cada vez que um geólogo coleta uma amostra de rocha de um furo, ele não joga a amostra antiga fora. Ele a armazena, cataloga e descreve: profundidade, data, composição, nome do responsável. Todas as amostras (camadas) formam um histórico completo da perfuração. Se for preciso reavaliar uma camada específica, basta ir ao registro e pegá-la.

É exatamente isso que um VCS faz com arquivos. Ele registra cada alteração, quem a fez, quando e por quê. Nada é perdido. Você pode "viajar no tempo" e ver exatamente como um arquivo era em qualquer ponto da história do projeto.

O sistema mais usado no mundo para isso se chama Git.

✅ Benefícios do Controle de Versão:
  • Histórico completo: Cada mudança é registrada permanentemente
  • Rastreabilidade: Saber quem fez o quê, quando e por quê
  • Reversibilidade: Voltar para qualquer versão anterior a qualquer momento
  • Colaboração segura: Múltiplas pessoas trabalhando simultaneamente
  • Branches: Experimentar sem afetar a versão principal
  • Backup distribuído: Cada colaborador tem uma cópia completa

Módulo 2: A Solução - Conhecendo as Ferramentas

Aula 3: O "Armazém Central" e as "Cópias Locais" (Repositórios)

O Git funciona com um modelo "distribuído". O que isso significa?

  • Repositório Remoto: Existe um "arquivo técnico central" na sede da mineradora. Este é o nosso Repositório Remoto (no GitHub, por exemplo). Ele contém a versão oficial e completa de todos os documentos do projeto.
  • Repositório Local: Quando um engenheiro precisa trabalhar em uma planta, ele não a edita diretamente no arquivo central. Ele solicita uma cópia autenticada e a leva para sua mesa. Essa cópia é o seu Repositório Local.

Ele pode fazer todas as anotações e rascunhos que quiser em sua cópia local, sem medo de atrapalhar o trabalho dos outros ou de estragar o documento oficial. O Git garante que a cópia local dele contenha todo o histórico do projeto até aquele momento.

Diferenças entre sistemas Centralizados e Distribuídos

Característica Sistema Centralizado (ex: SVN) Sistema Distribuído (ex: Git)
Histórico Apenas no servidor central Cópia completa em cada máquina
Trabalho offline Limitado Totalmente funcional
Velocidade Depende da rede Operações locais são instantâneas
Backup Ponto único de falha Cada clone é um backup completo
Branches Pesadas e lentas Leves e rápidas

Aula 4: Apresentando o Git e o GitHub

É importante não confundir os dois:

  • Git: É o sistema, a ferramenta, o "método de trabalho" que você instala no seu computador para controlar as versões dos arquivos. Ele é o motor por trás de tudo. Foi criado por Linus Torvalds em 2005 para gerenciar o desenvolvimento do kernel Linux.
  • GitHub: É uma plataforma online, um site, que serve como o "arquivo técnico central". É o lugar onde hospedamos nossos repositórios remotos para que toda a equipe possa colaborar. Pense no GitHub como a "sala de projetos online", onde todos se encontram para discutir e integrar suas mudanças.

Existem outras plataformas como o GitHub, por exemplo, GitLab e Bitbucket, mas todas usam o Git por baixo dos panos.

💡 Analogia prática:
  • Git é como o Microsoft Word instalado no seu computador
  • GitHub é como o OneDrive ou Google Drive onde você armazena e compartilha documentos

Módulo 3: Na Prática - Colaborando em um Projeto

Aula 5: O Mapa do Projeto - Entendendo o GitFlow

Antes de colocarmos a mão na massa, precisamos entender o "mapa de tráfego" do nosso projeto. Não criamos branches (ramificações) de qualquer maneira. Usamos um modelo de organização chamado GitFlow.

Analogia: Pense no GitFlow como as rotas e procedimentos de uma mina. Existem vias principais para o transporte pesado (versões estáveis), vias de acesso para desenvolvimento de novas áreas (funcionalidades) e desvios rápidos para reparos emergenciais (correções). Cada veículo (desenvolvedor) sabe exatamente qual rota seguir para não causar acidentes ou bloqueios.

O GitFlow define um conjunto de branches com propósitos específicos:

  • main: A branch principal. Ela representa a versão oficial, estável e em produção. Ninguém trabalha diretamente aqui. Ela só recebe o código que já foi testado e aprovado. É a nossa "planta mestra" arquivada.
  • develop: A branch de desenvolvimento. É aqui que todas as novas funcionalidades são integradas. Ela é o "canteiro de obras" principal, onde o trabalho do dia a dia acontece.
  • Branches de Funcionalidade (feature/...): São criadas a partir da develop. É aqui que você trabalha.
    • Exemplo: feature/relatorio-geologico-automatico
  • Branches de Lançamento (release/...): Quando a develop tem funcionalidades suficientes para uma nova versão, cria-se uma branch de release. Nela, apenas se fazem pequenos ajustes e correções finais antes do lançamento oficial.
  • Branches de Correção (hotfix/...): Se um bug crítico é encontrado na versão em produção (main), uma branch de hotfix é criada a partir da main para uma correção urgente.

Para você, colaborador, o mais importante é: seu trabalho sempre começará em uma branch feature (ou fix, docs, etc.) que nasce da develop.

✅ Nota importante sobre nomenclatura: A branch principal deve ser chamada de main. Em repositórios mais antigos, você pode encontrar o nome master, mas essa é uma nomenclatura antiga que está sendo substituída.

Aula 6: Fazendo sua cópia e sugerindo mudanças (Fork e Branch)

Você nunca trabalha diretamente na versão oficial. O processo é seguro e se divide em etapas:

1. Fork (Bifurcação)

O primeiro passo é criar um "fork" do repositório oficial.

Analogia: "Fork" é como criar sua própria cópia de segurança de todo o projeto em sua área pessoal dentro do GitHub. É como se você dissesse: "Vou pegar o arquivo mestre inteiro e levar uma cópia para o meu departamento, onde tenho total controle". Este "fork" ainda é um repositório remoto, mas é seu.

2. Clone (Clonar)

Depois de fazer o fork, você precisa baixar essa cópia para o seu computador:

$ git clone https://github.com/SEU-USUARIO/nome-do-projeto.git $ cd nome-do-projeto

3. Branch (Ramificação)

Agora, dentro do seu projeto, você cria uma "branch".

Analogia: Uma "branch" é como usar uma folha de papel vegetal sobre a planta original para desenhar uma sugestão. Você não risca o documento principal. Você cria uma "ramificação" temporária para trabalhar na sua ideia.

Exemplo de criação de branch:

$ git checkout develop $ git checkout -b feature/adicionar-mapa-de-risco

Esse método garante que o projeto principal (main branch) permaneça sempre estável e funcional.

Aula 7: Salvando e Descrevendo seu Trabalho (Commit e Conventional Commits)

Enquanto trabalha na sua branch, você precisa salvar seu progresso. Cada "ponto de salvamento" no Git é chamado de commit.

Analogia: Um "commit" é como preencher uma ficha de alteração para cada mudança significativa que você faz. Em vez de apenas salvar, você salva e descreve o que mudou e por quê.

Conventional Commits

Uma boa mensagem de commit é fundamental. Para padronizar isso, usamos um padrão chamado Conventional Commits. Ele formata a mensagem para que tanto humanos quanto máquinas possam entendê-la. A estrutura é:

<tipo>[escopo opcional]: <descrição>

[corpo opcional]

[rodapé opcional]

Tipos mais comuns:

  • feat: Uma nova funcionalidade
  • fix: Correção de um bug
  • docs: Alterações apenas na documentação
  • style: Mudanças que não afetam o significado do código (espaços, formatação)
  • refactor: Mudança no código que não corrige bug nem adiciona funcionalidade
  • test: Adição ou correção de testes
  • chore: Mudanças em ferramentas, configurações, dependências

Exemplos práticos para o seu contexto:

$ git add . $ git commit -m "feat(requisitos): adiciona critério de aceitação para relatório de perfuração" $ git commit -m "fix(documentacao): corrige link quebrado no manual do usuário" $ git commit -m "docs(readme): melhora a descrição do projeto e como executá-lo"

Fazer commits pequenos e bem descritos cria um histórico limpo e compreensível.

✅ Boas práticas para commits:
  • Faça commits pequenos e frequentes
  • Cada commit deve representar uma unidade lógica de mudança
  • Use o imperativo na descrição: "adiciona" ao invés de "adicionado"
  • A primeira linha deve ter no máximo 50 caracteres
  • Se necessário, adicione um corpo explicativo após uma linha em branco

Aula 8: Enviando suas sugestões para o time (Push e Pull Request)

Seu trabalho está pronto na sua branch local. Como você o apresenta para a equipe?

1. Push (Empurrar)

Primeiro, você faz um "push".

Analogia: "Push" é o ato de pegar a sua folha de papel vegetal com o desenho (sua branch) e levá-la até a "sala de projetos online" (seu repositório fork no GitHub), deixando-a visível para outros.
$ git push origin feature/adicionar-mapa-de-risco

2. Pull Request (Pedido de Integração)

Este é o passo mais importante da colaboração.

Analogia: O "Pull Request" (PR) é o seu pedido formal. Você vai até a equipe e diz: "Pessoal, aqui está minha sugestão. Por favor, revisem meu trabalho. Se estiver tudo certo, podemos integrá-lo (fazer o merge) à planta oficial".

Dentro do Pull Request no GitHub, a equipe pode:

  • Ver exatamente o que você alterou (linhas adicionadas em verde, removidas em vermelho)
  • Deixar comentários em linhas específicas do código
  • Solicitar ajustes ou melhorias
  • Aprovar a integração
  • Fazer o merge (integração) para a branch develop
✅ Boas práticas para Pull Requests:
  • Dê um título claro e descritivo
  • Descreva o que foi feito e por quê
  • Referencie issues relacionadas (ex: "Closes #123")
  • Mantenha o PR focado em uma única funcionalidade ou correção
  • Responda aos comentários dos revisores de forma construtiva
  • Mantenha seu PR atualizado com a branch de destino

Aula 9: Entendendo as Versões do Software (Versionamento Semântico - SemVer)

Quando um Pull Request é aprovado e integrado, ele pode gerar uma nova versão do software. Mas o que significa "versão 2.5.1"? Para isso, usamos o Versionamento Semântico (SemVer).

O formato é MAJOR.MINOR.PATCH (Ex: 1.4.2).

Componente Quando incrementar Exemplo no contexto de mineração
MAJOR (1.x.x) Mudanças que quebram a compatibilidade com versões anteriores Mudar o sistema de coordenadas de um mapa. Os dados antigos não são mais compatíveis com o novo formato. Requer atenção de todos.
MINOR (x.4.x) Novas funcionalidades adicionadas de forma retrocompatível Adicionar um novo tipo de relatório ao sistema. Os relatórios antigos continuam funcionando, mas agora há uma opção nova.
PATCH (x.x.2) Correções de bugs retrocompatíveis Corrigir um erro de cálculo em um relatório existente. A funcionalidade é a mesma, mas agora está correta.

Os Conventional Commits que vimos antes ajudam a automatizar isso! Se um commit é do tipo feat, o sistema sabe que deve incrementar a versão MINOR. Se for fix, incrementa a PATCH.

Exemplos de evolução de versões:

  • 1.0.0 → 1.0.1: Correção de bug (PATCH)
  • 1.0.1 → 1.1.0: Nova funcionalidade (MINOR)
  • 1.9.5 → 2.0.0: Mudança que quebra compatibilidade (MAJOR)
💡 Versão 0.x.x: Versões que começam com 0 (ex: 0.5.2) indicam que o software ainda está em desenvolvimento inicial e pode ter mudanças significativas a qualquer momento.

Criando seu Primeiro Repositório no GitHub

Agora que você entende os conceitos, vamos ao passo a passo prático para criar um repositório no GitHub.

Passo 1: Acesse o GitHub

  1. Acesse https://github.com
  2. Faça login na sua conta (ou crie uma conta gratuita se ainda não tiver)

Passo 2: Criar um Novo Repositório

  1. Clique no botão "+" no canto superior direito
  2. Selecione "New repository"
  3. Preencha as informações:
    • Repository name: Nome do seu projeto (ex: meu-primeiro-projeto)
    • Description: Breve descrição do que é o projeto
    • Public/Private: Escolha se o repositório será público ou privado
    • Initialize with README: Marque esta opção para criar um arquivo README.md inicial
    • .gitignore: Escolha um template se aplicável (ex: Python, Node, etc.)
    • License: Escolha uma licença se for um projeto open source
  4. Clique em "Create repository"

Passo 3: Clonar o Repositório para sua Máquina

Agora que o repositório foi criado no GitHub, você precisa cloná-lo para trabalhar localmente:

$ git clone https://github.com/SEU-USUARIO/meu-primeiro-projeto.git $ cd meu-primeiro-projeto

Passo 4: Fazer Alterações e Enviar

# Criar ou editar arquivos $ echo "# Meu Primeiro Projeto" > README.md # Adicionar arquivos ao staging $ git add . # Fazer commit $ git commit -m "docs: atualiza README com descrição do projeto" # Enviar para o GitHub $ git push origin main
✅ Pronto! Suas alterações agora estão no GitHub e visíveis para toda a equipe.

Comandos Git Essenciais para o Trabalho Local

Aqui estão os comandos mais importantes que você usará no dia a dia ao trabalhar com sua cópia local do repositório.

Configuração Inicial

# Configurar seu nome (aparecerá nos commits) $ git config --global user.name "Seu Nome" # Configurar seu e-mail $ git config --global user.email "seu.email@empresa.com" # Configurar branch padrão como 'main' $ git config --global init.defaultBranch main

Trabalhando com Branches

# Listar todas as branches $ git branch # Criar uma nova branch $ git branch feature/nova-funcionalidade # Mudar para uma branch $ git checkout feature/nova-funcionalidade # Criar e mudar para uma nova branch (atalho) $ git checkout -b feature/nova-funcionalidade # Deletar uma branch local $ git branch -d feature/antiga-funcionalidade

Verificando o Status e Histórico

# Ver o status dos arquivos (modificados, não rastreados, etc.) $ git status # Ver o histórico de commits $ git log # Ver o histórico de forma compacta $ git log --oneline # Ver diferenças entre arquivos modificados $ git diff

Salvando Alterações (Staging e Commit)

# Adicionar um arquivo específico ao staging $ git add arquivo.txt # Adicionar todos os arquivos modificados $ git add . # Fazer commit das alterações $ git commit -m "feat: adiciona nova funcionalidade" # Adicionar e fazer commit em um único comando (apenas arquivos já rastreados) $ git commit -am "fix: corrige bug no cálculo"

Sincronizando com o Repositório Remoto

# Baixar alterações do repositório remoto (sem fazer merge) $ git fetch origin # Baixar e integrar alterações da branch remota $ git pull origin develop # Enviar commits locais para o repositório remoto $ git push origin feature/minha-branch # Enviar e definir a branch remota como upstream (primeira vez) $ git push -u origin feature/minha-branch

Desfazendo Alterações

# Descartar alterações em um arquivo (antes do add) $ git checkout -- arquivo.txt # Remover arquivo do staging (depois do add, antes do commit) $ git reset HEAD arquivo.txt # Desfazer o último commit (mantendo as alterações) $ git reset --soft HEAD~1 # Desfazer o último commit (descartando as alterações) $ git reset --hard HEAD~1
⚠️ Atenção: Comandos como git reset --hard descartam alterações permanentemente. Use com cuidado!

Resolvendo Conflitos

Conflitos acontecem quando duas pessoas editam a mesma parte de um arquivo. O Git marca essas áreas:

<<<<<<< HEAD
Sua versão do código
=======
Versão do código de outra pessoa
>>>>>>> branch-name

Para resolver:

  1. Abra o arquivo e edite manualmente, escolhendo qual versão manter
  2. Remova os marcadores (<<<<<<<, =======, >>>>>>>)
  3. Adicione o arquivo resolvido: git add arquivo.txt
  4. Finalize o merge: git commit

Controle de Versão: Arquivos de Texto vs. Formatos Ricos

Nem todos os arquivos se beneficiam igualmente do controle de versão. Entender as diferenças é fundamental para trabalhar de forma eficiente.

Arquivos de Texto Puro (Plain Text)

São arquivos que contêm apenas caracteres legíveis, sem formatação especial. O Git funciona perfeitamente com eles.

Exemplos:

  • Código-fonte (.py, .js, .java, .c, .cpp, etc.)
  • Markdown (.md)
  • HTML, CSS, XML, JSON
  • Arquivos de configuração (.conf, .ini, .yaml, .env)
  • Scripts de shell (.sh, .bash)
  • Arquivos de texto simples (.txt)
✅ Vantagens com Git:
  • Diff linha por linha: O Git mostra exatamente quais linhas foram alteradas
  • Merge inteligente: Pode combinar alterações de diferentes pessoas automaticamente
  • Histórico legível: Você pode ver o conteúdo de qualquer versão anterior
  • Tamanho pequeno: Arquivos de texto ocupam pouco espaço
  • Resolução de conflitos: Conflitos podem ser resolvidos manualmente de forma clara

Arquivos Binários e Formatos Ricos

São arquivos que contêm dados em formato não legível diretamente por humanos. O Git pode armazená-los, mas com limitações significativas.

Exemplos:

  • Documentos do Microsoft Office (.docx, .xlsx, .pptx)
  • PDFs (.pdf)
  • Imagens (.png, .jpg, .gif, .svg)
  • Vídeos (.mp4, .avi, .mov)
  • Áudio (.mp3, .wav)
  • Arquivos compactados (.zip, .rar, .tar.gz)
  • Executáveis (.exe, .dll, .so)
⚠️ Limitações com Git:
  • Sem diff visual: O Git não consegue mostrar o que mudou dentro do arquivo
  • Sem merge automático: Se duas pessoas editarem, você terá que escolher uma versão completa
  • Tamanho do repositório: Cada versão do arquivo é armazenada integralmente, aumentando muito o tamanho
  • Histórico opaco: Não é possível ver o conteúdo de versões antigas sem baixá-las
  • Conflitos difíceis: Resolver conflitos significa escolher um arquivo inteiro

Tabela Comparativa

Característica Arquivos de Texto (.md, .py, .txt) Arquivos Binários (.docx, .pdf, .xlsx)
Visualização de diferenças ✅ Linha por linha, muito claro ❌ Apenas "arquivo mudou"
Merge automático ✅ Git combina alterações inteligentemente ❌ Impossível, escolha uma versão
Tamanho do repositório ✅ Pequeno, apenas diferenças são armazenadas ❌ Grande, cada versão completa é armazenada
Histórico navegável ✅ Pode ver qualquer linha em qualquer momento ❌ Precisa baixar o arquivo inteiro
Resolução de conflitos ✅ Edição manual clara e precisa ❌ Escolher versão A ou B inteira
Colaboração simultânea ✅ Múltiplas pessoas, mesma área ❌ Última pessoa a fazer push "ganha"

Recomendações Práticas

✅ Use Git para:

  • Todo código-fonte
  • Documentação em Markdown
  • Arquivos de configuração
  • Scripts e automações
  • Dados em formatos texto (CSV, JSON, XML)

⚠️ Evite ou tenha cuidado com:

  • Documentos do Office: Considere usar Google Docs ou Office 365 para colaboração em tempo real
  • Imagens grandes: Use serviços de armazenamento de mídia (S3, CDN) e referencie no código
  • Vídeos e áudio: Nunca coloque no Git. Use serviços especializados
  • Arquivos compilados: Adicione ao .gitignore para não versionar

💡 Alternativas para Documentos Colaborativos:

Tipo de Documento Melhor Ferramenta Por quê
Documentação técnica Markdown no Git Versionamento perfeito, fácil de revisar
Documentos de texto colaborativo Google Docs, Notion Edição simultânea em tempo real
Planilhas Google Sheets, Excel Online Colaboração em tempo real, histórico de versões
Apresentações Google Slides, PowerPoint Online Edição colaborativa, comentários
Diagramas Mermaid (texto), Draw.io Mermaid é texto puro, versionável

Caso Especial: Markdown vs. Word

Para documentação técnica, o Markdown é vastamente superior ao Word quando usado com Git:

✅ Markdown (.md):
  • Texto puro, totalmente versionável
  • Diff e merge funcionam perfeitamente
  • Pode ser convertido para HTML, PDF, Word
  • Leve e rápido
  • Funciona em qualquer editor
⚠️ Word (.docx):
  • Arquivo binário (XML compactado)
  • Git não mostra diferenças úteis
  • Merge impossível
  • Aumenta muito o tamanho do repositório
  • Requer software específico para abrir

Conclusão: Para documentação que será versionada com Git, sempre prefira Markdown. Para documentos colaborativos com formatação rica, use ferramentas online como Google Docs.
